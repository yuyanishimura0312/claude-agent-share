## 設計原則（このコマンド自体のベストプラクティス）

本チームは以下の実証済み原則に基づいて設計されている:
- **オーケストレーター・ワーカーパターン**: Anthropic推奨の中核アーキテクチャ（単一エージェント比で90%性能向上の実績）
- **2-5エージェント、各5-6タスク**: 実運用テスト済みの最適範囲。5人超はオーバーヘッドが利点を相殺
- **仕様の品質 = 出力の品質**: Phase 1のスコープ定義に最大の注意を払う
- **ハンドオフの構造化**: エージェント間の情報受け渡しは構造化フォーマットで行い、コンテキスト損失を防ぐ
- **CMOフレームワーク**: リアリスト評価の「文脈-メカニズム-成果」構成で「なぜ機能するか」を解明
- **GRADEシステム**: エビデンスの確度を4段階（高/中/低/非常に低い）で明示

## 実行パイプライン

### Phase 1: スコープ定義（リーダー直接実行）

テーマを分析し、以下を明確にする:
1. **調査対象の定義**: 「ベストプラクティス」が指すものを具体的に定義（手法か、プロセスか、設計原則か、ツール選定か）
2. **リサーチクエスチョン**: 3-5つの核心的な問いを設定
   - 「何が」ベストプラクティスとされているか
   - 「なぜ」それがベストとされるか（エビデンス）
   - 「どのような条件下で」有効か（適用範囲）
   - 「どう実装するか」（実行ステップ）
   - 「何に注意すべきか」（アンチパターン・落とし穴）
3. **調査の深度**:
   - quick: 概要把握（3エージェント各3クエリ）
   - standard: 標準調査（3エージェント各5-8クエリ）← デフォルト
   - deep: 徹底調査（3エージェント各10+クエリ、追加検証あり）

### Phase 2: 並列リサーチ（3エージェント同時起動）

**重要: 3つのAgentツールを1つのメッセージで同時に呼び出すこと。**

#### Agent A: Webリサーチャー
プロンプトに含める指示:
- WebSearchで日本語・英語の両方で検索（最低5クエリ）
- 対象: 業界レポート、テック企業ブログ、カンファレンス発表、専門メディア
- 有望なページはWebFetchで全文取得
- 各情報源の信頼性を5段階で評価
- 出力形式: 「発見事項」「情報源」「信頼性評価」のリスト

#### Agent B: 学術リサーチャー
プロンプトに含める指示:
- Semantic Scholar（search_papers, get_paper）で論文検索
- Paper Search（search_arxiv, search_google_scholar等）も併用
- 高被引用論文を優先
- 出力形式: 「論文名」「著者」「年」「主要な知見」「引用数」のリスト

#### Agent C: 事例リサーチャー
プロンプトに含める指示:
- WebSearchで成功事例・失敗事例・ケーススタディを検索
- 「best practice + [テーマ] + case study」「[テーマ] lessons learned」等で検索
- 企業規模・業界・コンテキストを記録
- 出力形式: 「事例名」「コンテキスト」「実践内容」「結果」「教訓」のリスト

### Phase 3: 構造解析（Agentで実行）

Phase 2の3つの結果を統合し、構造解析エージェントに渡す。

プロンプトに含める指示:
- **CMO（Context-Mechanism-Outcome）分析**: 各ベストプラクティスについて
  - Context: どのような状況・前提条件で有効か
  - Mechanism: なぜ機能するか（因果メカニズム）
  - Outcome: どのような結果をもたらすか
- **パターン抽出**: 複数ソースに共通する原則・パターンを特定
- **階層構造の整理**: 原則 → 戦略 → 戦術 → 具体的手法の階層で整理
- **適用条件マトリクス**: 各プラクティスの適用条件を整理
- **アンチパターンの特定**: やってはいけないこと、よくある失敗パターン

### Phase 4: 統合・品質検証（リーダー直接実行）

1. **三角検証**: 3つのリサーチ結果を突き合わせ
   - 複数ソースで裏付けられた知見 → 「確度:高」
   - 単一ソースのみ → 「確度:中」（要追加調査マーク）
   - ソース間で矛盾 → 両論併記
2. **GRADE評価**（各プラクティスのエビデンス品質を4段階で判定）:
   - 高: 複数の高品質ソースで裏付け、一貫した結果
   - 中: 中品質ソース、またはソース間でやや矛盾
   - 低: 限定的なエビデンス、間接的な関連のみ
   - 非常に低い: 専門家意見や理論的推論のみ
3. **批判的レビュー**:
   - 生存者バイアスがないか
   - 特定の文脈に依存しすぎていないか（普遍性 vs 文脈依存性）
   - 出版バイアス（成功事例のみが報告される傾向）がないか
   - 西村の文脈（小規模組織、コンサルティング、人類学研究）で適用可能か
4. **実行可能性評価**: 各プラクティスの実行難易度・コスト・効果を評価

### Phase 5: レポート作成

以下の構成でレポートを作成:

---

# [テーマ] ベストプラクティス構造解析レポート

## エグゼクティブサマリー（300-500字）
最も重要な発見と推奨事項を端的に。

## 1. 調査概要
- 調査スコープ、リサーチクエスチョン、調査方法

## 2. ベストプラクティスの全体像
- 文章（地の文）で、テーマにおけるベストプラクティスの全体構造を描写
- 原則レベルの知見を中心に

## 3. 構造解析（パターンカタログ形式）
各主要プラクティスを以下のパターン形式で記述:
- **名前**: プラクティスの簡潔な名称
- **問題**: どのような課題を解決するか
- **文脈（Context）**: どのような前提条件・状況で有効か
- **メカニズム（Mechanism）**: なぜ機能するか（因果プロセス、エビデンス付き）
- **成果（Outcome）**: どのような結果をもたらすか
- **実装ステップ**: 具体的な手順
- **注意点**: 落とし穴・アンチパターン・トレードオフ
- **エビデンス確度**: GRADE評価（高/中/低/非常に低い）

## 4. 適用条件マトリクス
プラクティスごとの適用条件・効果・難易度を表形式で整理

## 5. アンチパターン集
よくある失敗パターンとその回避方法

## 6. 推奨フレームワーク
調査結果を統合した実行可能なフレームワーク:
- ステップバイステップの実行手順
- 判断基準・チェックリスト
- 優先順位付け

## 7. 結論と次のアクション
- 即実行可能なアクション（3つ以内）
- 追加調査が必要な領域

## 情報源一覧
- 参照したURL・論文を列挙

- 「〜と思われます」ではなく、根拠を示す
